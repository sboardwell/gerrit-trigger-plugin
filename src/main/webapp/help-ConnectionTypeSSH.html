The traditional connection method where Gerrit events are received via SSH stream-events connection.

How it works:

  • Jenkins establishes a persistent SSH connection to the Gerrit server
  • Gerrit streams events in real-time using the gerrit stream-events command
  • The same SSH connection is used to send build results back to Gerrit (review comments, votes)

Requirements:

  • SSH access to the Gerrit server must be enabled
  • Jenkins must have SSH key authentication configured for the Gerrit user
  • The Gerrit user must have appropriate permissions to stream events
  • Network connectivity on the SSH port (typically 29418) must be available

Benefits:

  • Real-time: Events are delivered immediately with minimal latency
  • Reliable: Persistent connection with automatic reconnection on failure
  • Bidirectional: Same connection used for receiving events and sending build results
  • Proven: The original and most widely tested connection method

Considerations:

  • Requires maintaining a persistent SSH connection (may be blocked by some firewalls/proxies)
  • May require special firewall rules for SSH access
  • Connection drops will temporarily interrupt event delivery until reconnection

Recommended for: Most deployments, especially when Jenkins and Gerrit are in the same network or when SSH access is readily available.
